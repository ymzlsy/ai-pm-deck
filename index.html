Tool Deep-dive

关键工具：飞书 CLI

一句话：让 AI 能直接读写飞书里的数据，不用你手动复制粘贴

不需要打开飞书 App

在终端里一行命令就能抓到群消息、文档内容、日程——AI Agent 也能自己调用

信息直接变成结构化数据

群里聊了 200 条消息，CLI 一键导出成 JSON/Markdown，AI 直接分析

按项目隔离上下文

不像智能体把所有群混在一起——一个项目一个工作区，信息不串

能跟其他工具串起来

CLI 输出的数据可以直接喂给 Claude Code、Codex，形成自动化链条

实际怎么用？——两个真实场景
场景1：抓群消息做方案 
$ lark-cli im +chat-messages-list \
  --chat-id oc_xxx --page-size 50
# 拉最近50条群消息
# 输出 JSON → 喂给 Claude 分析需求
场景2：搜索特定消息 
$ lark-cli im +messages-search \
  --query "电子班牌" --format pretty
# 跨群搜索关键词
# 找出所有讨论过这个话题的消息
跟飞书智能体是什么关系？

CLI — 做从 0 到 1 的事

抓数据 → 分析 → 出方案 → 写代码，适合个人深度工作

智能体 — 做流程自动化

自动整理纪要、定时汇总、团队协作流转，适合日常协同

两者互补：我用 CLI 做深度分析和方案产出，用智能体做日常通知和流程自动化。不是选一个，是配合用。

一条链 · 局部角度

Part 2 里看到的变化——自己写代码、自己做调研、自己出原型——其实有一个成熟的名字：Forward Deployed Engineer（FDE）。这不是一个新发明，而是硅谷已经跑通的岗位模式。

? FDE 是什么

把最懂业务+最能写代码的人，直接嵌入客户现场。不是出差调研，而是长期驻场，当场把客户问题变成可运行的解决方案。

判别标准：写出来的东西在客户那里跑生产，并且公司平台因此变得更厚。

📋 日常工作节奏

AM跟客户业务专家对话，挖掘真实需求
PM用 AI 工具写代码，当场出 Demo
EVE让客户试用 → 收集反馈 → 当晚迭代
+1d新版本上线，继续循环

需要什么能力

业务理解力
能跟客户业务专家平等对话
工程能力
能独立从 0 搭出可用系统
沟通力
技术/业务/管理三层都聊得通
抗压韧性
现场处理混乱、变更、政治

Mismatch

我们卖的不是标准软件

而是「设备 + 仿真 + 课程 + 师资 + 考核」一体的教学解决方案。

业务现实

  • 轨道交通约占 70%，单订单 200–300 万量级
  • 招投标驱动，定制化极高，几乎每单二次开发
  • 必须进现场：硬件、软件、教学法一起落地
  • 需求常说不清——老师、教务、甲方三方统合

组织现实

  • 仍是销售 → 沟通 → 产品 → 开发 → 验收的瀑布
  • 各工位 KPI 独立，信息在转述中衰减
  • 产品拿到手时，常已是「二手需求」
  • AI 工具上了，但协作结构还没变

结论：交付物比 Palantir 典型客户更复杂，却用更长的远程瀑布链路——结构与业务本质错配，这正是 FDE 模式在捷安「反而更贴切」的原因。
